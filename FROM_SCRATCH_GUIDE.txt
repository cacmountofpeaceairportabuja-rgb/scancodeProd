FROM SCRATCH - A PRODUCTION-STYLE AUTH BLUEPRINT (Spring Boot)
==============================================================

This document is a "build-any-auth-feature" guide.

It is intentionally explicit:
- folder structure
- what each annotation means
- what each class is responsible for
- why we choose certain patterns (security + scalability)

Reality check (no magic)
------------------------
"Auth" is not one feature; it is a set of features:
1) Authentication (login)
2) Authorization (roles/permissions, policies)
3) Account lifecycle (signup, email verification, password reset)
4) Security hardening (rate limits, secure tokens, session/JWT choices, logging)

The demo project you have already showcases many authorization patterns.
This guide fills in the missing "product auth" pieces (signup + forgot password + real email).


PART A - Choose your architecture first
---------------------------------------

1) Sessions vs JWT (you MUST decide)
------------------------------------
Option 1: Session-based (recommended for MVC / server-rendered apps)
- Pros: simplest, secure-by-default with Spring Security, easy logout, easy CSRF protection for forms.
- Cons: requires a shared session store when you scale horizontally (or sticky sessions).

Option 2: JWT-based (common for pure APIs)
- Pros: can work well for mobile + SPAs, avoids server session state.
- Cons: revocation is hard, refresh rotation is hard, many teams implement it unsafely.

This demo uses sessions (form login). That is a great default if your UI is server-rendered.

2) What goes in the Principal?
------------------------------
In production you usually want your principal to expose:
- stable userId (not only username)
- email (and whether verified)
- tenantId/orgId (if multi-tenant)
- roles and/or permissions

You can represent this as:
- Spring's default `User` (simple but limited), OR
- a custom `UserDetails` (common), OR
- a custom `Authentication` (advanced)

The demo uses a custom principal (`DemoUser`) and a small interface (`UserEmail`) so different principal types can share a common contract.


PART B - Recommended folder structure (clean, scalable)
-------------------------------------------------------

Avoid putting everything in one package.

Example package layout:

`com.yourapp`
  `auth`
    `config`         (SecurityFilterChain, PasswordEncoder, etc.) ✅ config
    `controller`     (login/signup/reset endpoints)✅ as userauth
    `dto`            (request/response objects) ✅ dto
    `model`          (entities + value objects) ✅ 
    `repo`           (JPA repositories)
    `service`        (business logic)
    `token`          (verification/reset token helpers)
  `email`
    `service`        (EmailSender abstraction + implementations)
  `web`
    `controller`     (non-auth controllers)

Why this split?
- It prevents "god classes".
- It makes it easy to test services without booting the whole app.
- It keeps security-sensitive logic (tokens, password changes) centralized.

PART C - The core Spring Security concepts you must master
----------------------------------------------------------

1) SecurityFilterChain
----------------------
This is where you define:
- which routes are public vs protected
- how login works (formLogin, oauth2Login, httpBasic, oneTimeTokenLogin)
- CSRF, sessions, and security headers

2) UserDetailsService (or AuthenticationProvider)
-------------------------------------------------
This is how Spring Security loads users.
- "classic": implement `UserDetailsService`
- "advanced": custom `AuthenticationProvider` when you need multiple login modes

3) GrantedAuthority (roles and permissions)
-------------------------------------------
- A "role" is an authority that starts with `ROLE_`.
- `hasRole("ADMIN")` checks for `ROLE_ADMIN`.
- Permissions can be authorities like `SHIPMENT_READ`, `SHIPMENT_WRITE`.

Best practice:
- Roles: broad categories (ADMIN, USER)
- Permissions: fine-grained actions (REFUND_APPROVE)

4) Where to put authorization rules
-----------------------------------
Use BOTH:
1) Request authorization (URL rules) for coarse route protection
2) Method authorization (`@PreAuthorize`) for business rules

Example:
- URL: `/admin/**` -> hasRole(ADMIN)
- Service: `approveRefund(...)` -> hasAuthority("REFUND_APPROVE") AND same tenant

Why DelegatingPasswordEncoder?
- It stores hashes like `{bcrypt}...`
- You can migrate encodings later without breaking existing users.

Step 2: User entity (JPA)
-------------------------
File: `src/main/java/.../auth/model/AppUser.java`

--- code ---
@Entity
@Table(name = "users")
public class AppUser {
  @Id
  @GeneratedValue(strategy = GenerationType.IDENTITY)
  private Long id;

  @Column(nullable = false, unique = true)
  private String username;

  @Column(nullable = false, unique = true)
  private String email;

  @Column(name = "password_hash", nullable = false)
  private String passwordHash;

  @Column(nullable = false)
  private boolean enabled; // email verified? (recommended)

  @ElementCollection(fetch = FetchType.EAGER)
  @CollectionTable(name = "user_roles", joinColumns = @JoinColumn(name = "user_id"))
  @Column(name = "role", nullable = false)
  private Set<String> roles = new HashSet<>();

  // getters/setters omitted for brevity
}
--- end ---

JPA annotations explained:
- `@Entity`: this class maps to a database table.
- `@Table`: controls the table name (recommended for clarity).
- `@Id`: primary key.
- `@GeneratedValue`: how the ID is generated.
- `@Column`: database constraints. Still validate in code too.
- `@ElementCollection`: stores simple collection values in a separate table.

Step 4: DatabaseUserDetailsService
----------------------------------
File: `src/main/java/.../auth/service/DatabaseUserDetailsService.java`

--- code ---
@Service
public class DatabaseUserDetailsService implements UserDetailsService {
  private final UserRepository userRepository;

  public DatabaseUserDetailsService(UserRepository userRepository) {
    this.userRepository = userRepository;
  }

  @Override
  public UserDetails loadUserByUsername(String username) throws UsernameNotFoundException {
    var user = userRepository.findByUsername(username)
      .orElseThrow(() -> new UsernameNotFoundException("User not found"));

    if (!user.isEnabled()) {
      throw new DisabledException("Email not verified");
    }

    var authorities = user.getRoles().stream()
      .map(r -> r.startsWith("ROLE_") ? r : "ROLE_" + r)
      .map(SimpleGrantedAuthority::new)
      .toList();

    return new org.springframework.security.core.userdetails.User(
      user.getUsername(),
      user.getPasswordHash(),
      authorities
    );
  }
}
--- end ---

Why not return your entity directly?
-----------------------------------
Keeping a clean separation prevents accidental persistence-side effects and lets you change DB schema without rewriting security code.

Step 5: Security configuration
------------------------------
File: `src/main/java/.../auth/config/SecurityConfig.java`

--- code ---
@Configuration
@EnableWebSecurity
@EnableMethodSecurity
public class SecurityConfig {

  @Bean
  SecurityFilterChain appSecurity(HttpSecurity http) throws Exception {
    return http
      .authorizeHttpRequests(auth -> auth
        .requestMatchers("/", "/auth/**", "/css/**", "/js/**").permitAll()
        .requestMatchers("/admin/**").hasRole("ADMIN")
        .anyRequest().authenticated()
      )
      .formLogin(form -> form
        .loginPage("/auth/login")         // optional custom login page
        .defaultSuccessUrl("/dashboard", true)
      )
      .logout(logout -> logout
        .logoutUrl("/auth/logout")
        .logoutSuccessUrl("/")
      )
      .build();
  }
}
--- end ---

Security annotations explained:
- `@EnableWebSecurity`: enables Spring Security web support.
- `@EnableMethodSecurity`: enables `@PreAuthorize`, `@PostAuthorize`, etc.

Step 6: Signup service
----------------------
Rules you should enforce:
- validate username/email/password (Bean Validation)
- normalize email (lowercase)
- store password hash only
- assign default role USER
- enabled=false until email verified (recommended)

File: `src/main/java/.../auth/service/UserService.java`

--- code ---
@Service
public class UserService {
  private final UserRepository userRepository;
  private final PasswordEncoder passwordEncoder;

  public UserService(UserRepository userRepository, PasswordEncoder passwordEncoder) {
    this.userRepository = userRepository;
    this.passwordEncoder = passwordEncoder;
  }

  @Transactional
  public AppUser register(String username, String email, String rawPassword) {
    var normalizedEmail = email.toLowerCase(Locale.ROOT);

    if (userRepository.existsByUsername(username)) {
      throw new IllegalArgumentException("Username already taken");
    }
    if (userRepository.existsByEmail(normalizedEmail)) {
      throw new IllegalArgumentException("Email already used");
    }

    var user = new AppUser();
    user.setUsername(username);
    user.setEmail(normalizedEmail);
    user.setPasswordHash(passwordEncoder.encode(rawPassword));
    user.setEnabled(false);
    user.getRoles().add("USER");

    return userRepository.save(user);
  }
}
--- end ---

Service annotations explained:
- `@Service`: marks business logic bean.
- `@Transactional`: ensures DB changes commit/rollback together.


PART E - Forgot password / reset password (secure design)
---------------------------------------------------------

This is where many apps make security mistakes.
Follow these rules:
1) Do not store raw reset tokens in the database.
2) Store a hash of the token (like password hashing).
3) Tokens must expire.
4) Rate-limit and avoid user enumeration.
   - Always respond: "If the email exists, we sent a link."

Suggested model:
- `password_reset_tokens` table:
  - user_id
  - token_hash
  - expires_at
  - used_at
  - created_at

Flow:
1) User submits email.
2) If user exists, generate random token (SecureRandom).
3) Store token hash + expiry, email raw token in a link.
4) User opens link and submits new password + token.
5) Server hashes token, finds record, checks expiry/used_at.
6) If valid, update passwordHash and mark token used.

Step 1: Secure token helper
---------------------------
File: `src/main/java/.../auth/token/SecureToken.java`

--- code ---
public final class SecureToken {
  private static final SecureRandom RNG = new SecureRandom();

  private SecureToken() {}

  public static String generateUrlSafe(int bytes) {
    byte[] data = new byte[bytes];
    RNG.nextBytes(data);
    return Base64.getUrlEncoder().withoutPadding().encodeToString(data);
  }

  public static String sha256Hex(String raw) {
    try {
      MessageDigest md = MessageDigest.getInstance("SHA-256");
      byte[] digest = md.digest(raw.getBytes(StandardCharsets.UTF_8));
      return HexFormat.of().formatHex(digest);
    } catch (NoSuchAlgorithmException e) {
      throw new IllegalStateException(e);
    }
  }
}
--- end ---

Step 4: PasswordResetService
----------------------------
File: `src/main/java/.../auth/service/PasswordResetService.java`

--- code ---
@Service
public class PasswordResetService {
  private final UserRepository userRepository;
  private final PasswordResetTokenRepository tokenRepository;
  private final PasswordEncoder passwordEncoder;
  private final EmailSender emailSender;

  public PasswordResetService(UserRepository userRepository,
                              PasswordResetTokenRepository tokenRepository,
                              PasswordEncoder passwordEncoder,
                              EmailSender emailSender) {
    this.userRepository = userRepository;
    this.tokenRepository = tokenRepository;
    this.passwordEncoder = passwordEncoder;
    this.emailSender = emailSender;
  }

  @Transactional
  public void requestReset(String email) {
    var normalizedEmail = email.toLowerCase(Locale.ROOT);
    var userOpt = userRepository.findByEmail(normalizedEmail);
    if (userOpt.isEmpty()) {
      return; // avoid enumeration
    }

    var rawToken = SecureToken.generateUrlSafe(32);
    var tokenHash = SecureToken.sha256Hex(rawToken);

    var token = new PasswordResetToken();
    token.setUser(userOpt.get());
    token.setTokenHash(tokenHash);
    token.setCreatedAt(Instant.now());
    token.setExpiresAt(Instant.now().plus(Duration.ofMinutes(15)));
    tokenRepository.save(token);

    var link = "https://yourapp/reset-password?token=" + rawToken;
    emailSender.send(normalizedEmail, "Reset your password", "Click: " + link);
  }

  @Transactional
  public void resetPassword(String rawToken, String newPassword) {
    var tokenHash = SecureToken.sha256Hex(rawToken);
    var token = tokenRepository.findValid(tokenHash, Instant.now())
      .orElseThrow(() -> new IllegalArgumentException("Invalid token"));

    var user = token.getUser();
    user.setPasswordHash(passwordEncoder.encode(newPassword));
    userRepository.save(user);

    token.setUsedAt(Instant.now());
    tokenRepository.save(token);
  }
}
--- end ---

Why hash the reset token?
-------------------------
If your database leaks, attackers cannot use stored tokens because only hashes exist.

PART G - Common Spring annotations (short reference)
----------------------------------------------------

`@Configuration`
- Declares a configuration class that defines beans.

`@Bean`
- Registers a method's return value as a Spring bean.

`@Service`, `@Component`, `@Controller`, `@RestController`
- Stereotypes for component scanning and injection.

`@EnableWebSecurity`
- Enables Spring Security web integration.

`@EnableMethodSecurity`
- Enables method authorization annotations (Pre/PostAuthorize).

`@PreAuthorize("...")`
- Runs BEFORE the method; denies invocation if expression is false.

`@PostAuthorize("...")`
- Runs AFTER the method; can inspect the return value (returnObject).

`@Valid`
- Triggers Bean Validation on request objects.

`@NotBlank`, `@Email`, `@Size`
- Validation constraints for inputs.


PART H - Minimum security checklist (do not skip)
-------------------------------------------------

1) Passwords
- Use PasswordEncoder (bcrypt/argon2 via delegating encoder).
- Enforce minimum length (and optionally reject known weak passwords).

2) Reset tokens
- Use SecureRandom.
- Hash tokens in DB.
- Expire tokens quickly.
- Mark tokens used.
- Rate limit reset requests.

3) Sessions and cookies (session-based auth)
- Use HTTPS.
- Ensure cookies are HttpOnly + Secure.
- Set SameSite appropriately.
- Configure session timeout.

4) CSRF
- Keep CSRF enabled for browser forms.
- For pure JSON APIs, decide on CSRF strategy early (depends on whether you use cookies).

5) Audit logs
- Log login success/failure, password reset requested, password changed.
